Le 3 regole d'oro

  1. Contesto: linguaggio, framework, vincoli tecnici
  2. Obiettivo: cosa vuoi ottenere esattamente
  3. Formato: come vuoi l'output

Struttura idee vaghe in piani implementabili

Workflow spec-kit

1. /speckit.constitution

Definisci principi: quality, testing, performance

2. /speckit.specify

Descrivi COSA vuoi (user stories, requisiti)

3. /speckit.clarify

Chiarifica requisiti ambigui (opzionale)

Workflow spec-kit (continua)

4. /speckit.plan

Definisci COME: tech stack, architettura

5. /speckit.tasks

Genera task list ordinata con dipendenze

6. /speckit.implement

Esegue tutti i task in ordine

Tra ogni step: itera, raffina, valida

graph TD A[/speckit.constitution/] -->|Principi progetto| B[/speckit.specify/] B -->|Requisiti COSA| C{Tutto chiaro?} C -->|No| D[/speckit.clarify/] D --> C C -->|Sì| E[/speckit.plan/] E --> F[/speckit.tasks/] F --> G[/speckit.implement/]

Esempio concreto

# 1. constitution
/speckit.constitution
"""
Principi: code quality, test coverage >80%, performance
Tech vincolata: TypeScript, Prisma, PostgreSQL
No breaking changes su API esistenti
"""

# 2. specify
/speckit.specify
"Sistema di notifiche real-time per eventi utente.
Push notifications + email. Storia completa notifiche."

# 3. plan
/speckit.plan
"Stack: NestJS, Prisma, WebSocket, Bull queue, SendGrid"

Quando usare spec-kit

Usa spec-kit quando:

  • Feature complessa multi-file
  • Serve coordinazione team
  • Requisiti da validare prima di codare

Non serve per:

  • Quick fix o hotfix
  • Refactoring singola funzione
  • Spike o prototipi veloci

Takeaways

Prompt efficaci:

contesto + obiettivo + vincoli + formato output

Itera sempre:

prima bozza, poi raffina, poi valida

spec-kit per progetti complessi:

constitution → specify → plan → tasks → implement
